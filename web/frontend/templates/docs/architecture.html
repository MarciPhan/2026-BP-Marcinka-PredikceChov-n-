Architektura systému

Metricord je distribuovaný systém navržený pro zpracování milionů událostí v reálném čase. Tato stránka podrobně popisuje všechny komponenty, jejich komunikaci a datový tok.

4. Tok dat (Data Flow)

Co se děje, když uživatel pošle zprávu na Discord serveru:

  1. Event Capture: Bot zachytí událost on_message přes Discord Gateway WebSocket.
  2. Pre-processing: Vypočítá se váha zprávy (délka, reply bonus) a XP. Spam check: cooldown 60s, duplicate detection.
  3. Redis Push (atomický):
    • PFADD do HyperLogLog pro DAU/MAU tracking.
    • ZADD do Sorted Setu pro uživatelské eventy (timestamped).
    • HINCRBY pro hodinové a denní počítadla zpráv.
    • HINCRBY pro heatmapu (den_v_týdnu × hodina).
  4. Dashboard Read: Při návštěvě webu FastAPI přečte agregovaná data z Redisu, aplikuje matematické modely (Markov, Kaplan-Meier) a renderuje Jinja2 šablonu s Chart.js grafy.

5. Background Tasks

Bot spouští dva periodické úlohy na pozadí:

Úloha Interval Účel
member_stats_task 10 sekund Aktualizuje online/total počet členů v Redisu, guild security metadata (verification level, MFA, explicit filter).
heartbeat_task 60 sekund Zapisuje timestamp do bot:heartbeat. Dashboard ho čte pro indikátor „Bot is UP/DOWN".

6. Redis datový model

Kompletní přehled klíčů z shared/keys.py:

Klíč (pattern) Typ Popis
hll:dau:{guild}:{YYYYMMDD} HyperLogLog Denní unikátní aktivní uživatelé. ~12 KB / den / guild.
stats:hourly:{guild}:{YYYYMMDD} Hash Počet zpráv za každou hodinu (field = „0"–„23").
stats:heatmap:{guild} Hash Aktivita po dnech a hodinách (field = „0:14" = pondělí 14:00).
stats:msglen:{guild} Hash Distribuce délky zpráv (field = bucket, value = count).
stats:total_msgs:{guild} String Kumulativní celkový počet zpráv.
events:msg:{guild}:{user} Sorted Set Timestampy zpráv uživatele (score = unix timestamp).
events:voice:{guild}:{user} Sorted Set Začátky/konce voice sezení uživatele.
events:action:{guild}:{user} Sorted Set Moderátorské akce (ban, kick, timeout, delete).
user:info:{user} Hash Cache uživatelských údajů (jméno, avatar, role).
presence:online:{guild} String (TTL: 60s) Aktuálně online členové. Zapisuje member_stats_task.
bot:heartbeat String Unix timestamp posledního heartbeatu bota.
bot:guilds Set Seznam guild IDs, kde je bot přítomen.

7. Matematické modely

Implementace v shared/models.py (třída CommunityModels):

Markovovy řetězce

  • 5 stavů: New(0), Active(1), Passive(2), Inactive(3), Churned(4).
  • calculate_markov_matrix() — počítá matici přechodů z pozorovaných dat.
  • predict_future_states() — predikce rozložení stavů po N krocích (dot product vektoru se maticí).

Kaplan-Meier (Survival Analysis)

  • calculate_survival_rate() — křivka přežití z dob přežití a cenzorovaných pozorování.
  • estimate_life_expectancy() — střední délka přežití (plocha pod survival křivkou).

8. Bezpečnostní model

Vrstva Mechanismus Detail
Autentizace Discord OAuth2 Nikdy nevidíme heslo uživatele. Token flow s PKCE.
Session itsdangerous signed cookies Kryptograficky podepsané session tokeny s TTL 24h.
API Bearer token + guild permission check Každý request ověřuje guild membership a Manage Server oprávnění.
Redis Interní síť (botnet) Redis nikdy nevystavěn na veřejný port. Komunikace jen přes Docker network.
Důležité pro vývojáře Bot a Dashboard mohou běžet na různých serverech. Jediným pojítkem je přístup k identické instanci Redis. Tato architektura umožňuje horizontální škálování — více botů zapisuje, jeden dashboard čte.
