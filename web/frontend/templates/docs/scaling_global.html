Global Scaling & High Availability

Průvodce škálováním Metricord pro velké komunity (50k+ členů) a zajištění vysoké dostupnosti pro produkční nasazení.

1. Redis Sentinel (High Availability)

Pro zajištění běhu i při výpadku hlavního databázového uzlu využijte Redis Sentinel. Ten automaticky zvolí nového Mastera, pokud hlavní uzel selže.

# sentinel.conf
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1

Topologie

Node Role Data
redis-master Master (read/write) Bot zapisuje sem
redis-replica-1 Replica (read-only) Dashboard čte odtud (snížení latence)
redis-replica-2 Replica (read-only) Záloha pro případ failoveru
sentinel-1, 2, 3 Monitor Sledují master, automatický failover

2. Redis Cluster (Sharding)

Pro servery s miliony eventů, kde se data nevejdou do RAM jednoho uzlu:

  • Data jsou automaticky rozdělena do 16,384 hash slotů napříč uzly.
  • Klíče s guild ID jako prefixem zajišťují, že data jednoho serveru jsou na stejném uzlu (hash tagy: {guild_id}).
  • Doporučení: 3+ master nodes, každý s 1 replikou.
# Vytvoření clusteru
redis-cli --cluster create \
    node1:6379 node2:6379 node3:6379 \
    node4:6379 node5:6379 node6:6379 \
    --cluster-replicas 1

3. Read/Write Splitting

Pro optimální výkon oddělte zapisující a čtecí klienty:

# .env pro bot (Write)
REDIS_URL=redis://redis-master:6379/0

# .env pro dashboard (Read)
REDIS_URL=redis://redis-replica:6379/0

Bot vždy zapisuje do Mastera, Dashboard čte z Repliky → nulový dopad analytiky na výkon bota.

4. Event Thinning

Pro servery s extrémně vysokou aktivitou (1M+ zpráv/měsíc) zvažte:

  • Voice sampling: Změna frekvence voice trackingu z 1 minuty na 5 minut v activity.py.
  • Message aggregation: Místo zápisu každé zprávy zvlášť, agregovat do 5minutových bucketů.
  • TTL na events: Nastavit expiraci starých Sorted Setů (events:*) po 90 dnech.
# Příklad TTL na event klíče
redis-cli EXPIRE "events:msg:guild_id:user_id" 7776000  # 90 dní

5. Multi-Node Deployment

Pro globální komunity s členy v různých časových zónách:

Region Komponenty Účel
EU-Central Redis Master + Bot Primary + Dashboard Hlavní uzel, všechna data
US-East Redis Replica + Dashboard Read-Only Nižší latence pro americké moderátory
Architektonický tip Pro globální škálování je klíčová latence k Redisu. Bot i Redis musí být v rámci stejného regionu a ideálně stejné VPC / privátní sítě. Dashboard repliky mohou být geograficky distribuované.

6. Monitoring velkých instalací

# Sledování Redis metriky v reálném čase
redis-cli --stat

# Detailní monitoring
redis-cli INFO all | grep -E "(used_memory_human|connected_clients|instantaneous_ops|keyspace)"

# Sledování pomalých příkazů
redis-cli SLOWLOG GET 10
